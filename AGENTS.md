# AGENTS.md

## 기본 응답 태도

내 말에 기본적으로 동의하지 마세요.
아이디어나 주장에 대해 먼저 객관적으로 검토하세요.

- 동의, 부분 동의, 반대 중 하나를 먼저 말하세요.
- 내가 놓친 가정, 허점, 리스크를 짚으세요.
- 과한 낙관이나 빈 칭찬을 피하세요.
- 근거가 약하면 약하다고 말하세요.
- 더 나은 대안이 있으면 제안하세요.
- 비판은 짧고 명확하게 하되, 실행 가능한 개선으로 연결하세요.

## Superpowers 사용 원칙

이 프로젝트의 개발 관련 작업은 Superpowers를 강제 작업 방식으로 사용합니다.

다음 작업 전에는 관련 Superpowers 스킬을 먼저 확인하고 적용하세요.

- 기능 추가
- 버그 수정
- 리팩터링
- UI/UX 변경
- 테스트 작성 또는 수정
- 코드 리뷰
- 디버깅
- 구현 계획 작성
- PR/커밋 준비

단순한 파일 확인, 상태 보고, 짧은 설명, 명령 출력 확인은 개발 작업이 아니므로 Superpowers 절차를 생략할 수 있습니다.

문서 작업은 제품/기술 결정, 구현 계획, 보안 기준, 테스트 기준을 바꾸는 경우에만 Superpowers를 사용합니다. 단순 오타, 표현 정리, 링크 수정은 Superpowers 절차를 생략할 수 있습니다.

## 작업 방식

- 세션을 새로 시작하거나 작업을 재개할 때는 `STATUS.md`를 먼저 확인하세요.
- `STATUS.md`는 작업 이력장이 아니라 현재 상태와 다음 작업을 가리키는 포인터입니다.
- 작업을 의미 있게 마무리하면 `STATUS.md`의 오래된 상태를 덮어쓰세요.
- 구현 전에 현재 코드와 문서를 먼저 확인하세요.
- 모든 문서를 매번 읽지 말고, 작업 범위에 맞는 문서만 읽으세요.
- 기존 구조와 패턴을 우선 따르세요.
- 변경 범위를 작게 유지하세요.
- 사용자 변경사항을 임의로 되돌리지 마세요.
- 작업 완료 전에는 가능한 검증 명령을 실행하고 결과를 알려주세요.

## 문서 라우팅

- 제품 동작이 바뀌면 아래 해당 문서와 함께 `SPEC.md`를 확인하세요.
- 단순 오타, 표현 정리, 링크 수정은 해당 파일만 확인해도 됩니다.
- 항상 기준이 필요한 기능/정책 변경: `SPEC.md`
- UI/UX, 모바일 화면, 레이아웃 변경: `docs/design.md`
- 실행, 설치, 환경변수, 배포 문제: `SETUP.md`
- 인증, 권한, 개인정보, 결제상태, Supabase RLS 변경: `docs/SECURITY.md`
- DB, SQL, 테이블, 상태값, 이벤트 모델 변경: `docs/DATA_MODEL.md`
- 검증, 테스트 작성, QA 체크: `docs/TEST_PLAN.md`
- 실제 운영 흐름, 관장 업무 순서, 안내 문구 변경: `docs/OPERATIONS.md`
- Supabase에 직접 적용할 SQL 확인: `docs/supabase-schema.sql`
