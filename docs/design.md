# 강동무에타이장 PT 회원 예약 MVP UI/UX 디자인 문서

## 1. 디자인 방향

- 현재 우선순위는 체육관 전체 관리앱이 아니라 PT 회원 예약 MVP다.
- 모바일 접속을 1순위로 둔다.
- 관장과 회원 모두 휴대폰에서 가장 자주 접속한다고 전제한다.
- UI는 모바일 화면에서 먼저 성립해야 하며, 태블릿/PC에서는 같은 구조를 넓게 확장한다.
- 현장 운영 도구이므로 장식보다 빠른 확인, 빠른 판단, 빠른 처리를 우선한다.
- 마케팅형 랜딩 페이지는 만들지 않는다. 첫 화면은 바로 사용할 수 있는 운영 화면이어야 한다.

## 2. 모바일 우선 기준

- 모든 주요 버튼과 탭은 최소 44px 이상의 터치 영역을 가진다.
- 모바일에서 페이지 전체 가로 스크롤은 생기면 안 된다.
- 가로 스크롤은 관리자 탭, 회원 선택 목록처럼 의도된 영역에만 허용한다.
- 카드 안의 텍스트는 줄바꿈되어야 하며, 버튼이나 상태 배지를 침범하면 안 된다.
- 모바일 첫 화면에서 `처리 필요`가 먼저 보이고, 이번 주 예약 상태 요약이 이어져야 한다.
- PC 전용 기능은 만들지 않는다. 모든 주요 기능은 모바일에서 완료 가능해야 한다.

## 3. 관리자 UX

- 관리자 모바일 첫 흐름은 `처리 필요 확인 -> 주간 요약 확인 -> 일정 탭 진입 -> 주간 날짜 strip 확인 -> 시간 목록에서 예약/취소/수업완료 처리`이다.
- 처리 필요 패널은 모바일에서 가장 먼저 노출한다.
- 처리 필요 항목은 종류, 회원명, 금액/횟수 등 판단에 필요한 최소 정보만 보여준다.
- 관리자 하단 탭은 `홈`, `일정`, `회원`, `설정`으로 둔다.
- 관리자 하단 탭 4개는 모바일과 데스크톱에서 같은 폭으로 균등 정렬한다.
- `홈`은 처리 필요를 먼저 보여주고 작은 7일 주간 요약을 함께 제공한다.
- `일정` 상세 화면은 기본적으로 주간 날짜 strip과 선택 날짜 시간 목록을 먼저 보여준다.
- `월` 보기는 월간 달력을 크게 펼치는 탐색 모드이고, `주` 보기는 주간 날짜 strip과 시간 목록 처리에 집중한다.
- 일정 상단 툴바는 `월/주` 보기 select와 `전체/PT/오전반/초등부/일반부` 일정 타입 select 2개만 둔다.
- PT는 별도 하단 탭이 아니라 통합 일정 안의 타입이다. 현재 단계에서 `전체`와 `PT`는 기존 PT 슬롯/예약을 보여주고, `오전반`, `초등부`, `일반부`는 빈 수업 일정 상태만 보여준다.
- 일반 수업 예약, 정원, 출석 액션은 이번 단계의 UI에 노출하지 않는다.
- `일정` 상세 화면은 주간 날짜 strip에서 날짜를 빠르게 고르고, 필요할 때 `월` 보기로 월간 달력을 펼쳐 탐색한다.
- 관리자 달력은 운영 처리 우선순위로 표시한다. 예약요청/취소요청 같은 처리 필요 날짜를 가장 먼저 알아볼 수 있어야 하며, 선택 날짜 시간 목록에는 시간 범위, 상태, 승인/거절/완료/차감 액션을 표시한다.
- 관리자와 회원의 선택 날짜 시간 목록은 `시간 범위 / 상태 / 필요한 액션`만 남겨 중복 설명 문장을 줄인다.
- 슬롯 상태는 `가능`, `요청중`, `확정`, `차단`을 명확히 구분한다.
- 액션 버튼은 아이콘 중심으로 두되, `title`/`aria-label`로 의미를 제공한다.
- `설정` 첫 화면은 메뉴 목록만 보여주고, 세부 입력은 메뉴 진입 후 표시한다.
- `운영 정책`은 `예약`, `취소`, `연장`, `재등록` 하위 메뉴를 한 번 더 거친다.

## 4. 회원 UX

- 회원 모바일 첫 흐름은 `내 PT 상태 확인 -> 월간 달력에서 날짜 선택 -> 시간 목록에서 예약 요청/취소 -> 내역/연장 상태 확인`이다.
- 회원 하단 탭은 `홈`, `예약`, `내역`으로 둔다.
- 회원 화면 상단은 compact header로 둔다. header에는 회원 이름, 승인 상태 pill, 회원 메뉴만 노출하며 이메일 fallback은 쓰지 않는다.
- 잔여횟수, 만료일, 결제상태, 재등록 상태는 회원 홈의 PT 요약에서 확인한다.
- 예약 화면은 달력에서 내 요청/확정 예약일과 예약 가능일을 먼저 구분할 수 있어야 한다.
- 회원 달력은 다른 회원의 예약 정보를 보여주지 않는다. 날짜 칸은 `내 예약`, `대기`, `가능 N`처럼 회원 본인에게 의미 있는 상태만 표시한다.
- 예약 가능 시간은 월간 달력 날짜 선택 후 해당 날짜의 시간 목록으로 제공한다. 다른 회원이 잡았거나 운영상 차단된 시간은 기본적으로 숨기고, 회원 본인의 예약 요청/확정/취소요청과 예약 가능한 시간만 보여준다.
- 회원의 확정 예약 취소 액션은 예약 탭의 선택 날짜 시간 목록에서 바로 처리할 수 있게 둔다.
- 내역 화면에서는 회원권 상세, 결제일/결제상태, 차감/연장/환불 이력, 연장 요청/승인 상태, 예약 이력을 확인할 수 있어야 한다.
- 회원이 예약 요청을 보낸 뒤에는 승인 대기 상태가 명확히 보여야 한다.
- 결제상태가 미납이거나 결제요청 상태면 회원 화면에서 숨기지 않고 명확히 표시한다.

## 5. 정보 구조

- 관리자 화면의 기본 구조:
  - 상단: 앱 이름, 관리자/회원 전환
  - 상태줄: 최근 처리 결과
  - 하단 탭: 홈, 일정, 회원, 설정
  - 홈: 처리 필요 패널, 주간 요약
  - 일정: compact 월/주 보기 select, 일정 타입 select, 주간 날짜 strip, 시간/상태/액션 중심 시간 목록, 월 보기 달력
  - 회원/PT권
  - 설정: PT 상품, 운영 정책, 안내 문구, CSV 내보내기
  - 운영 정책: 예약, 취소, 연장, 재등록
- 회원 화면의 기본 구조:
  - compact header: 회원 이름 / 승인 상태 / 회원 메뉴. 이름은 승인된 회원 연결 요청의 표시 이름을 우선하고, 없으면 이메일이 아닌 회원명 또는 `회원`을 쓴다.
  - 하단 탭: 홈, 예약, 내역
  - 홈: 내 PT 요약, 다음 예약
  - 예약: 내 요청/확정 예약일과 예약 가능일을 구분하는 월간 달력, 선택 날짜의 내 예약/예약 가능 시간 목록
  - 내역: 회원권 상세, 결제, 차감/연장/환불, 연장 요청/승인 상태, 예약 이력

회원 `예약` 화면은 이번 단계에서 PT 예약 UX를 유지한다. 이후 일반 수업 예약을 도입할 때는 회원 예약 화면 안에 `PT / 수업` 전환을 추가하는 방향으로 확장한다.

## 5.1 개발/테스트 전환 장치

- 현재 로컬 데모에서는 관리자/회원 화면 확인을 쉽게 하기 위해 상단 전환을 유지한다.
- 최종 Supabase Auth/RLS 연결 후에는 로그인 계정 권한에 따라 관리자 또는 회원 화면으로 분기한다.
- 회원 계정에 관리자 데이터와 액션이 노출되는 구조를 최종 제품으로 취급하면 안 된다.

## 6. 시각 규칙

- 기본 카드 반경은 8px 이하로 유지한다.
- Apple iOS 운영 UI 적용 범위에서는 최상위 그룹 표면만 16~18px 반경을 허용하고, 반복 행/슬롯/상태 항목은 10~14px 이하로 제한한다.
- 상태 배지는 색으로 빠르게 구분하되 텍스트 라벨을 반드시 함께 제공한다.
- 주요 상태 색상:
  - 정상/완료: 녹색 계열
  - 요청/미납/주의: 노란색 계열
  - 취소요청/위험: 붉은색 계열
  - 정보: 파란색 계열
- 한 가지 색상만 지배하는 화면은 피한다.
- 화면 섹션을 과도한 중첩 카드로 만들지 않는다.

## 7. 반응형 기준

- 기본 CSS는 모바일 레이아웃이다.
- `min-width: 721px`부터 태블릿/작은 PC 확장 레이아웃을 적용한다.
- `min-width: 1101px`부터 데스크톱 다열 레이아웃을 적용한다.
- 모바일에서는 하단 탭을 고정하고, 회원 선택 목록은 가로 스크롤 영역으로 둔다.
- 데스크톱 홈에서는 처리 필요 패널과 주간 요약을 나란히 두어 넓은 화면을 활용한다.
